# Client-Generated event_id for End-to-End Deduplication

Each Event Envelope carries a UUID v4 `event_id` generated by the client before transmission. PostgreSQL stores events with a unique constraint on `event_id`; the Batch Worker inserts with `ON CONFLICT DO NOTHING`. This achieves effectively-once storage despite at-least-once delivery from Redis Streams.

The client must generate the ID (not the server) because server-side assignment would issue a new ID on each retry, defeating deduplication entirely. Client-generated IDs mean a retried request carries the same `event_id`, so the duplicate is silently dropped by PostgreSQL regardless of whether it arrived via a client retry or a worker re-process from the stream.
